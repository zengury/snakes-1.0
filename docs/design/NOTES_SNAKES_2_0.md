# Snakes 2.0 — Design Notes（讨论沉淀）

本文件用于收录 PRD 之外的讨论结论与设计取舍，避免在开发过程中丢失上下文。

## 1) 主线优先（Golden Path First）

Snakes 2.0 的核心价值不在某个“字段设计/模块细节”，而在于把以下主线贯穿并稳定化：

1. Context：组装机器人身份/角色/约束/状态摘要
2. Think：LLM/策略产出下一步工具调用
3. Act：通过 CLI toolchain 执行动作（统一入口）
4. Verify：强制观察/验证（动作 != 成功）
5. Recover：失败后进入恢复路径（重试/换视角/诊断/降级）
6. Record：所有事实写入 EventLog（统一事实源）
7. Promote（可选）：沉淀验证器/恢复器为 skill（不优先沉淀成功脚本）

所有新增功能都必须能明确挂靠到以上某一步，否则延后。

## 2) 为什么用“密室逃脱”作为抽象

密室逃脱代表一类长周期具身任务：探索-验证-失败恢复反复发生，多模态切换频繁。
主流“更强模型+直接 API”容易在失败与复盘上爆炸；Snakes 通过 runtime 主线把系统能力固化。

## 3) CLI 工具链的独特优势

CLI 的意义不是“工具列表”，而是提供可组合的执行语义：
- 并发读（降低等待与 turn 数）
- 超时/取消/重试（系统级鲁棒性）
- 资源互斥与安全闸（硬约束）
- 运维工具（diagnosis）同样通过工具链接入，提升长周期稳定性

## 4) Failure-first 的地位

Failure-first 不是创新点本身，而是主线能跑通的前提：
- 没有统一 outcome/failure taxonomy，就无法统计、无法恢复、无法比较。
- 重点是：不要沉迷字段；只保留支撑主线所需的最小结构。

## 5) 技能沉淀策略

优先沉淀：验证器与恢复器（跨场景迁移价值高）。
谨慎沉淀：单场景成功路径（往往过拟合）。

## 6) 机器人本地小模型（预留）

预留本地小模型的位置，用于：
- 低延迟/离线的观测压缩与粗分类
- 快速故障分类（system/perception/manipulation）
- 轻量恢复决策的候选生成

2.0 仅预留 provider/接口，不强行实现大一统。
